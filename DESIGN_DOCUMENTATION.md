## 1. Database Schema Design

### Entity-Relationship Model

#### Book Entity
The primary entity in our library management system.

**Table Name:** `library_book`

| Field Name       | Data Type      | Constraints          | Description                           |
|------------------|----------------|----------------------|---------------------------------------|
| id               | BigInteger     | PRIMARY KEY, AUTO    | Unique identifier for each book       |
| title            | VARCHAR(200)   | NOT NULL             | Title of the book                     |
| author           | VARCHAR(200)   | NOT NULL             | Author's full name                    |
| cover_pic        | VARCHAR(100)   | NULL, BLANK          | Path to book cover image              |
| isbn             | VARCHAR(13)    | UNIQUE, NOT NULL     | 13-character ISBN number              |
| description      | TEXT           | NOT NULL             | Detailed book description             |
| category         | VARCHAR(50)    | NOT NULL             | Book category/genre                   |
| published_date   | DATE           | NOT NULL             | Publication date                      |
| available_copies | INTEGER        | DEFAULT 1, NOT NULL  | Number of available copies            |

**Indexes:**
- Primary Key: id
- Unique Index: isbn
- Index: title (for ordering)

**Sample Data:**
```sql
INSERT INTO library_book VALUES
(1, 'The Great Gatsby', 'F. Scott Fitzgerald', NULL, '9780743273565',
 'A novel set in the Jazz Age...', 'Classic Fiction', '1925-04-10', 5),
(2, 'To Kill a Mockingbird', 'Harper Lee', NULL, '9780061120084',
 'A gripping tale of racial injustice...', 'Classic Fiction', '1960-07-11', 3);
```

### Database Relationships (Future Extensions)

#### Potential Future Entities:
1. **Member/User Entity** - For library members
   - id, username, email, phone, membership_date, status

2. **Borrowing Entity** - For tracking book loans
   - id, book_id (FK), member_id (FK), borrow_date, due_date, return_date, status

3. **Reservation Entity** - For book reservations
   - id, book_id (FK), member_id (FK), reservation_date, status

## Design Patterns Used

### 1. Model-Template-View (MTV)
Django's architectural pattern separating concerns:
- **Model:** Data layer (Book model)
- **Template:** Presentation layer (HTML templates)
- **View:** Business logic (view functions)

### 2. Don't Repeat Yourself (DRY)
- Reusable Book model with __str__ method
- Centralized settings configuration
- Template inheritance structure (future)

### 3. Convention Over Configuration
- Standard Django project structure
- Default migration naming
- Automatic admin interface generation

### 4. Object-Relational Mapping (ORM)
- Django ORM abstracts database operations
- No raw SQL required for CRUD operations
- Database-agnostic design (easy to switch databases)

## Security Considerations

### Implemented Security Measures

1. **Secret Key Protection**
   - SECRET_KEY stored in .env file
   - .env file excluded from version control

2. **Database Credentials**
   - All credentials in environment variables
   - No hardcoded passwords in code

3. **CSRF Protection**
   - Django's CSRF middleware enabled
   - All forms protected by CSRF tokens

4. **SQL Injection Prevention**
   - Django ORM prevents SQL injection
   - Parameterized queries used automatically

5. **XSS Protection**
   - Django templates auto-escape HTML
   - Clickjacking protection enabled

6. **File Upload Security**
   - ImageField validates file types
   - Upload directory isolated (media/)
   - File size limits can be configured

### Recommended Future Enhancements

1. **Authentication**
   - Implement user registration and login
   - Role-based access control (Admin, Librarian, Member)
   - Session timeout configuration

2. **Input Validation**
   - Custom validators for ISBN format
   - Date range validation for published_date
   - Minimum/maximum copy validation

3. **Production Security**
   - DEBUG=False in production
   - ALLOWED_HOSTS configuration
   - HTTPS enforcement
   - Database connection over SSL
   - Regular security updates

## Database Design Principles

### Normalization
The Book model follows 3NF (Third Normal Form):
- Each field contains atomic values
- No partial dependencies on primary key
- No transitive dependencies

### Data Integrity
- **Primary Key:** Auto-incrementing id
- **Unique Constraint:** ISBN field
- **Foreign Keys:** (Future: Member, Borrowing tables)
- **Check Constraints:** available_copies >= 0 (model validation)

### Performance Optimization
- **Indexing:** Primary key on id, unique index on isbn
- **Ordering:** Default ordering by title for efficient queries
- **Charset:** utf8mb4 for full Unicode support including emojis
